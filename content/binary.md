# 一.组会
64位实例：
```python
payload = offset+pop_rdi+puts_got+puts_plt+main_sym
```
## 1. 彻底理解 GOT、PLT、sym 与动态链接
### 1.1 三个概念的身份卡片

|名称|存储区域|本质|例子（gyctf_2020_borrowstack）|
|---|---|---|---|
|**`main` (sym)**|代码段 (.text)|可执行代码|`0x400626`：真正的 `main` 函数起始地址|
|**`puts_plt`**|代码段 (.plt)|一小段跳转代码|`0x4004e0`：`jmp [puts_got]` 等指令|
|**`puts_got`**|数据段 (.got.plt)|一个 8 字节的指针变量|`0x601018`：里面存放 `puts` 的真实地址|
### 1.2辨析
**重要**：

- `main` 是**真正的函数**，包含函数序言（`push rbp; mov rbp, rsp; sub rsp, 0x60` 等）。
    
- `puts_plt` 不是函数，它是一个**跳板**，用来间接调用 `puts`。
    
- `puts_got` 是一个**变量**，它的值在运行时被动态链接器写入。
## 2.从编译视角理解
> 编译时 vs 运行时：谁决定地址？

### 2.1 编译时（静态）

- 编译器不知道 `puts` 会加载到哪个地址（因为 libc 的基址由 ASLR 随机决定）。
    
- 所以编译器生成代码时，不直接写 `call 0x7f...`，而是：
    
    - 在 `.plt` 段生成一段代码（`puts_plt`），里面包含 `jmp [puts_got]`。
        
    - 在 `.got.plt` 段预留一个槽位（`puts_got`），初始值设为某个占位地址（通常是 `puts_plt` 中的下一条指令，用于触发动态链接器）。
        
    - 在调用 `puts` 的地方，生成 `call puts_plt`。
        

### 2.2运行时（动态）

- 程序启动时，`puts_got` 尚未填入真实地址。
    
- **第一次**执行 `call puts_plt` 时：
    
    1. `puts_plt` 执行 `jmp [puts_got]`，此时 `puts_got` 指向 `puts_plt` 内部的一段解析代码。
        
    2. 解析代码调用动态链接器 `ld-linux.so`。
        
    3. 动态链接器查找 `puts` 在内存中的真实地址（根据 libc 基址 + 固定偏移），将该地址写入 `puts_got`。
        
    4. 然后跳转到真实的 `puts` 函数执行。
        
- **第二次**及以后执行 `call puts_plt` 时：
    
    - `jmp [puts_got]` 直接跳转到已写入的真实 `puts` 地址，不再经过解析。
        

**所以**：

- **`main_sym`** 的地址在编译时就固定了（未开 PIE 时）。
    
- **`puts_plt`** 的地址也在编译时固定。
    
- **`puts_got`** 的地址固定（数据段），但**它的内容**在运行时才确定。

## 3这个动态编译的原理和我们ret2libc的攻击原理一样
### 3.1 编译
- 编译时got表加载出libc基地址，真实地址为libc+offset
- call puts_plt即跳转到jmp [puts_got],注意了puts_got是一个地址单元，在got表中，而[puts_got]是这个地址单元上的内容，即函数的真实地址
### 3.2 ret2libc等等
- 涉及到`libc`库就打出libc地址，再加上偏移量
- 注意了如果是给了libc的elf文件，那就是libc_base = addr_puts - libc.sym[puts]了
## 4.解析这段payload
```python
payload2 = offset + p64(pop_rdi) + p64(puts_got) + p64(puts_plt) + p64(main)
```
让我们拆解执行过程：

1. **`p64(pop_rdi)`** 将 `pop rdi; ret` 的地址放在栈上。
    
2. **`p64(puts_got)`** 是 `pop rdi` 弹出的值，即 `0x601018` 这个数字。
    
3. **`p64(puts_plt)`** 是 `ret` 后要跳转的地址，即 `0x4004e0`（`puts_plt` 的入口）。
    
4. 执行 `puts_plt` 时，它会执行 `jmp [puts_got]`，而 `puts_got` 此时已经存放了 `puts` 的真实地址（因为在之前可能已经调用过 `puts`，或者第一次调用时动态链接器会解析）。
    
5. 所以 `puts` 被调用，它的参数 `rdi` 是 `0x601018`，即 `puts_got` 的地址。
    
6. `puts` 从地址 `0x601018` 开始读取字符串，直到遇到 `\x00`。而 `0x601018` 处存放的是 `puts` 的真实地址（8 字节），该地址通常以 `0x7f` 开头，后面跟着 libc 地址的高位，`puts` 会把这个地址当作字符串输出（直到遇到 `\x00`，地址中一般没有 `\x00` 所以会输出 6~8 字节）。
    
7. 我们通过 `p.recv(6)` 接收这 6 字节地址，补全为 8 字节，得到 `puts` 在 libc 中的真实地址。
    

**关键**：`puts_got` 作为参数传给 `puts`，让 `puts` 打印它**自己的 GOT 表项内容**，从而泄露真实地址。

                              ↓
┌────────────────────────────────────────────────────────────────┐
│ 运行时（第一次调用）                                            │

│ 2. 此时 0x601018 内容 = 0x4004e6（解析代码）                  │

                              ↓
┌────────────────────────────────────────────────────────────────┐
│ 运行时（后续调用）                                              │
├────────────────────────────────────────────────────────────────┤
│ 1. call 0x4004e0  → jmp [0x601018]                           │
│ 2. 此时 0x601018 内容 = 0x7f1234567890（真实地址）            │
│ 3. 直接跳转到 puts 函数                                        │
